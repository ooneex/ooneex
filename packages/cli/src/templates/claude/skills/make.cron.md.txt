# Make Cron Class

Generate a new cron class and its test file using the `make:cron` CLI command, then complete the generated code with proper implementation.

## Coding Conventions

- Always explicitly show visibility on class methods and properties (`private`, `public`, `protected`)
- Always use arrow functions when possible, but NOT for class methods (class methods use regular method syntax)
- End type names with the `Type` suffix (e.g., `ExceptionStackFrameType`, `ServiceDataType`)
- Start interface names with the `I` prefix (e.g., `IException`, `IService`, `IAiChat`)
- For Entity class, if property is optional, add `null` to its type.

## Steps

### 1. Run the generator

Run the following command to scaffold the cron class and test files:

```bash
bunx @ooneex/cli@latest make:cron --name=<name> --module=<module>
```

Where `<name>` is the name provided by the user. The `--module` option is optional — if provided, files are generated under `modules/<module>/` instead of the project root. The command will generate:
- `src/cron/<Name>Cron.ts` - The cron class file (or `modules/<module>/src/cron/<Name>Cron.ts` with `--module`)
- `tests/cron/<Name>Cron.spec.ts` - The test file (or `modules/<module>/tests/cron/<Name>Cron.spec.ts` with `--module`)

### 2. Read the generated files

Read both generated files to understand the scaffolded code.

### 3. Complete the cron class

Edit `src/cron/<Name>Cron.ts` to complete the implementation:

- Set the appropriate cron schedule in `getTime()` (e.g., `"every 5 minutes"`, `"every 1 hours"`, `"every 30 seconds"`)
- Set the timezone in `getTimeZone()` if needed, or keep `null` for server timezone
- Implement the `handler()` method with the actual cron job logic
- Inject any required dependencies via the constructor

The generated class structure follows this pattern:

```typescript
import type { TimeZoneType } from "@ooneex/country";
import { Cron, type CronTimeType, decorator } from "@ooneex/cron";

@decorator.cron()
export class <Name>Cron extends Cron {
  public getTime(): CronTimeType {
    // Examples: "every 5 minutes", "every 1 hours", "every 30 seconds"
    return "every 1 hours";
  }

  public getTimeZone(): TimeZoneType | null {
    // Return null to use server timezone, or specify a timezone like "Europe/Paris"
    return null;
  }

  public async handler(): Promise<void> {
    // Implement your cron handler logic here
    // console.log("<Name>Cron handler executed");
  }
}
```

### 4. Complete the test file

Edit `tests/cron/<Name>Cron.spec.ts` to add meaningful tests beyond the scaffolded ones:

- Keep the existing scaffolded tests (class name, getTime, getTimeZone, handler methods)
- Add tests relevant to the specific cron class behavior

The generated test structure follows this pattern:

```typescript
import { describe, expect, test } from "bun:test";
import { <Name>Cron } from "@/cron/<Name>Cron";

describe("<Name>Cron", () => {
  test("should have class name ending with 'Cron'", () => {
    expect(<Name>Cron.name.endsWith("Cron")).toBe(true);
  });

  test("should have 'getTime' method", () => {
    expect(<Name>Cron.prototype.getTime).toBeDefined();
    expect(typeof <Name>Cron.prototype.getTime).toBe("function");
  });

  test("should have 'getTimeZone' method", () => {
    expect(<Name>Cron.prototype.getTimeZone).toBeDefined();
    expect(typeof <Name>Cron.prototype.getTimeZone).toBe("function");
  });

  test("should have 'handler' method", () => {
    expect(<Name>Cron.prototype.handler).toBeDefined();
    expect(typeof <Name>Cron.prototype.handler).toBe("function");
  });
});
```

### 5. Register the cron job in the module

Add the new cron job to the module's `cronJobs` array in `src/<PascalModuleName>Module.ts` (e.g., `src/BookModule.ts` for the `book` module):

```typescript
import type { ModuleType } from "@ooneex/module";
import { <Name>Cron } from "./cron/<Name>Cron";

export const <PascalModuleName>Module: ModuleType = {
  controllers: [],
  entities: [],
  middlewares: [],
  cronJobs: [<Name>Cron],
  events: [],
};
```

The module file uses PascalCase naming: `<PascalModuleName>Module.ts` with export `<PascalModuleName>Module`.

If the module already has other cron jobs registered, append the new cron job to the existing `cronJobs` array and add the import alongside existing imports.

### 6. Lint and format

Run linting and formatting on the generated files:

```bash
bunx biome check --fix src/cron/<Name>Cron.ts tests/cron/<Name>Cron.spec.ts
```
