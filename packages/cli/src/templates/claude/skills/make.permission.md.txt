# Make Permission Class

Generate a new permission class and its test file using the `make:permission` CLI command, then complete the generated code with proper implementation.

## Coding Conventions

- Always explicitly show visibility on class methods and properties (`private`, `public`, `protected`)
- Always use arrow functions when possible, but NOT for class methods (class methods use regular method syntax)
- End type names with the `Type` suffix (e.g., `ExceptionStackFrameType`, `ServiceDataType`)
- Start interface names with the `I` prefix (e.g., `IException`, `IService`, `IAiChat`)

## Steps

### 1. Run the generator

Run the following command to scaffold the permission class and test files:

```bash
bunx @ooneex/cli@latest make:permission --name <name> --module <module>
```

Where `<name>` is the name provided by the user. The `--module` option is optional — if provided, files are generated under `modules/<module>/` instead of the project root. The command will generate:
- `src/permissions/<Name>Permission.ts` - The permission class file (or `modules/<module>/src/permissions/<Name>Permission.ts` with `--module`)
- `tests/permissions/<Name>Permission.spec.ts` - The test file (or `modules/<module>/tests/permissions/<Name>Permission.spec.ts` with `--module`)

### 2. Read the generated files

Read both generated files to understand the scaffolded code.

### 3. Complete the permission class

- Implement the `allow()` method with permission rules using `this.ability.can()`
- Implement the `setUserPermissions()` method with role-based permission logic

### 4. Complete the test file

Edit `tests/permissions/<Name>Permission.spec.ts` to add meaningful tests beyond the scaffolded ones:

- Keep the existing scaffolded tests (class name, extends Permission, allow, setUserPermissions methods)
- Add tests relevant to the specific permission rules

The generated test structure follows this pattern:

```typescript
import { describe, expect, test } from "bun:test";
import { Permission } from "@ooneex/permission";
import { <Name>Permission } from "@/permissions/<Name>Permission";

describe("<Name>Permission", () => {
  test("should have class name ending with 'Permission'", () => {
    expect(<Name>Permission.name.endsWith("Permission")).toBe(true);
  });

  test("should extend Permission", () => {
    const permission = new <Name>Permission();
    expect(permission).toBeInstanceOf(Permission);
  });

  test("should have 'allow' method", () => {
    expect(<Name>Permission.prototype.allow).toBeDefined();
    expect(typeof <Name>Permission.prototype.allow).toBe("function");
  });

  test("should have 'setUserPermissions' method", () => {
    expect(<Name>Permission.prototype.setUserPermissions).toBeDefined();
    expect(typeof <Name>Permission.prototype.setUserPermissions).toBe("function");
  });
});
```

### 5. Register the permission in the module

Add the new permission to the module's `permissions` array in `src/<PascalModuleName>Module.ts` (e.g., `src/BookModule.ts` for the `book` module):

```typescript
import type { ModuleType } from "@ooneex/module";
import { <Name>Permission } from "./permissions/<Name>Permission";

export const <PascalModuleName>Module: ModuleType = {
  controllers: [],
  entities: [],
  permissions: [<Name>Permission],
  middlewares: [],
  cronJobs: [],
  events: [],
};
```

The module file uses PascalCase naming: `<PascalModuleName>Module.ts` with export `<PascalModuleName>Module`.

If the module already has other permissions registered, append the new permission to the existing `permissions` array and add the import alongside existing imports.

### 6. Lint and format

Run linting and formatting on the generated files:

```bash
bunx biome check --fix src/permissions/<Name>Permission.ts tests/permissions/<Name>Permission.spec.ts
```
