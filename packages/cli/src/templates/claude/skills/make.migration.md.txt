# Make Migration

Generate a new migration file and its test file using the `make:migration` CLI command, then complete the generated code with proper implementation.

## Coding Conventions

- Always explicitly show visibility on class methods and properties (`private`, `public`, `protected`)
- Always use arrow functions when possible, but NOT for class methods (class methods use regular method syntax)
- End type names with the `Type` suffix (e.g., `ExceptionStackFrameType`, `ServiceDataType`)
- Start interface names with the `I` prefix (e.g., `IException`, `IService`, `IAiChat`)
- For Entity class, if property is optional, add `null` to its type.
- For dependencies, use `inject` from `@ooneex/container`. For example:

```typescript
import { inject } from "@ooneex/container";
import { decorator, type ISeed } from "@ooneex/seeds";

@decorator.seed()
export class BookSeed implements ISeed {
  constructor(
    @inject(BookRepository)
    private readonly repository: BookRepository,
  ) {}
}
```

## Steps

### 1. Run the generator

Run the following command to scaffold the migration file and test files:

```bash
bunx @ooneex/cli@latest make:migration --module=<module>
```

The `--module` option is optional — if provided, files are generated under `modules/<module>/` instead of the project root. The command will generate:
- `src/migrations/Migration<version>.ts` - The migration class file (or `modules/<module>/src/migrations/Migration<version>.ts` with `--module`)
- `tests/migrations/Migration<version>.spec.ts` - The test file (or `modules/<module>/tests/migrations/Migration<version>.spec.ts` with `--module`)

It will also:
- Generate a `migrations.ts` root export file in the migrations directory

### 2. Read the generated files

Read both generated files to understand the scaffolded code.

### 3. Complete the migration

Edit `src/migrations/Migration<version>.ts` to implement:

- The `up` method with the schema changes (create tables, add columns, create indexes, etc.)
- The `down` method with the reverse operations to undo the migration

### 4. Complete the test file

Edit `tests/migrations/Migration<version>.spec.ts` to add meaningful tests beyond the scaffolded ones:

- Keep the existing scaffolded tests (class name, up, down, getVersion, getDependencies methods)
- Add tests relevant to the specific migration behavior

The generated test structure follows this pattern:

```typescript
import { describe, expect, test } from "bun:test";
import { Migration<version> } from "@/migrations/Migration<version>";

describe("Migration<version>", () => {
  test("should have class name starting with 'Migration'", () => {
    expect(Migration<version>.name.startsWith("Migration")).toBe(true);
  });

  test("should have 'up' method", () => {
    expect(Migration<version>.prototype.up).toBeDefined();
    expect(typeof Migration<version>.prototype.up).toBe("function");
  });

  test("should have 'down' method", () => {
    expect(Migration<version>.prototype.down).toBeDefined();
    expect(typeof Migration<version>.prototype.down).toBe("function");
  });

  test("should have 'getVersion' method", () => {
    expect(Migration<version>.prototype.getVersion).toBeDefined();
    expect(typeof Migration<version>.prototype.getVersion).toBe("function");
  });

  test("should have 'getDependencies' method", () => {
    expect(Migration<version>.prototype.getDependencies).toBeDefined();
    expect(typeof Migration<version>.prototype.getDependencies).toBe("function");
  });
});
```

### 5. Lint and format

Run linting and formatting on the generated files:

```bash
bunx biome check --fix src/migrations/Migration<version>.ts tests/migrations/Migration<version>.spec.ts
```
