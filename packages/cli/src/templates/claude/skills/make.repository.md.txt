# Make Repository Class

Generate a new repository class and its test file using the `make:repository` CLI command, then complete the generated code with proper implementation.

## Steps

### 1. Run the generator

Run the following command to scaffold the repository class and test files:

```bash
bunx @ooneex/cli@latest make:repository --name <name> --module <module>
```

Where `<name>` is the name provided by the user. The `--module` option is optional — if provided, files are generated under `modules/<module>/` instead of the project root. The command will generate:
- `src/repositories/<Name>Repository.ts` - The repository class file (or `modules/<module>/src/repositories/<Name>Repository.ts` with `--module`)
- `tests/repositories/<Name>Repository.spec.ts` - The test file (or `modules/<module>/tests/repositories/<Name>Repository.spec.ts` with `--module`)

### 2. Read the generated files

Read both generated files to understand the scaffolded code.

### 3. Complete the repository class

Edit `src/repositories/<Name>Repository.ts` to complete the implementation:

- Verify the entity import path matches the actual entity location
- Adjust the `find` method's search fields (default searches `name` with `ILike`)
- Customize relations loading in `findOne`/`findOneBy` if needed

#### Adding methods

Look at the entity's fields, relations, and business context to determine if custom domain-specific methods are needed. For example:
- A `SessionRepository` might need `revokeSession(sessionId: string)` and `revokeAllUserSessions(userId: string)`
- A `NotificationRepository` might need `markAsRead(id: string)` and `markAllAsRead(userId: string)`
- Entities with status fields may need methods like `archive(id: string)` or `activate(id: string)`

Read related entities, services, or actions in the module to understand what operations the repository should support, then add the appropriate methods.

#### Removing methods

Remove scaffolded methods that don't make sense for the entity's context:
- Remove `createMany`/`updateMany` if the entity is always managed individually (e.g., user profiles, settings)
- Remove `delete` if the entity uses soft deletes only (use a custom `softDelete` or `archive` method instead)
- Remove `find` if the entity is only ever accessed by ID or specific criteria (e.g., singleton config entities)
- Remove `count` if there's no use case for counting records of this entity

The generated class structure follows this pattern:

```typescript
import { inject } from "@ooneex/container";
import type { ITypeormDatabase } from "@ooneex/database";
import { decorator } from "@ooneex/repository";
import type { FilterResultType } from "@ooneex/types";
import type { FindManyOptions, FindOptionsWhere, Repository, SaveOptions, UpdateResult } from "typeorm";
import { ILike } from "typeorm";
import { <Name>Entity } from "../entities/<Name>Entity";

@decorator.repository()
export class <Name>Repository {
  constructor(
    @inject("database")
    private readonly database: ITypeormDatabase,
  ) {}

  public async open(): Promise<Repository<<Name>Entity>> {
    return await this.database.open(<Name>Entity);
  }

  public async close(): Promise<void> {
    await this.database.close();
  }

  public async find(
    criteria: FindManyOptions<<Name>Entity> & { page?: number; limit?: number; q?: string },
  ): Promise<FilterResultType<<Name>Entity>> {
    // ... pagination and search logic
  }

  public async findOne(id: string): Promise<<Name>Entity | null> { ... }
  public async findOneBy(criteria: FindOptionsWhere<<Name>Entity>): Promise<<Name>Entity | null> { ... }
  public async create(entity: <Name>Entity, options?: SaveOptions): Promise<<Name>Entity> { ... }
  public async createMany(entities: <Name>Entity[], options?: SaveOptions): Promise<<Name>Entity[]> { ... }
  public async update(entity: <Name>Entity, options?: SaveOptions): Promise<<Name>Entity> { ... }
  public async updateMany(entities: <Name>Entity[], options?: SaveOptions): Promise<<Name>Entity[]> { ... }
  public async delete(criteria: FindOptionsWhere<<Name>Entity> | FindOptionsWhere<<Name>Entity>[]): Promise<UpdateResult> { ... }
  public async count(criteria?: FindOptionsWhere<<Name>Entity> | FindOptionsWhere<<Name>Entity>[]): Promise<number> { ... }
}
```

### 4. Complete the test file

Edit `tests/repositories/<Name>Repository.spec.ts` to add meaningful tests beyond the scaffolded ones:

- Keep scaffolded tests for methods that remain in the repository (remove tests for methods that were removed)
- Add tests for any custom domain-specific methods added to the repository
- Add tests relevant to the specific repository behavior

### 5. Lint and format

Run linting and formatting on the generated files:

```bash
bunx biome check --fix src/repositories/<Name>Repository.ts tests/repositories/<Name>Repository.spec.ts
```
