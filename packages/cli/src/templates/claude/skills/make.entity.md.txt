# Make Entity Class

Generate a new entity class and its test file using the `make:entity` CLI command, then complete the generated code with proper implementation.

## Steps

### 1. Run the generator

Run the following command to scaffold the entity class and test files:

```bash
bunx @ooneex/cli@latest make:entity --name <name> --table-name <table_name>
```

Where `<name>` is the name provided by the user. The `--table-name` option is optional — if omitted, it defaults to the snake_case pluralized form of the name (e.g., `UserProfile` becomes `user_profiles`). The command will generate:
- `src/entities/<Name>Entity.ts` - The entity class file
- `tests/entities/<Name>Entity.spec.ts` - The test file

### 2. Read the generated files

Read both generated files to understand the scaffolded code.

### 3. Complete the entity class

Edit `src/entities/<Name>Entity.ts` to complete the implementation:

- Add entity-specific columns with appropriate TypeORM decorators (`@Column`)
- Add relations if needed (`@ManyToOne`, `@OneToMany`, `@ManyToMany`, etc.)
- Remove any scaffolded columns that are not relevant to the entity
- Adjust column types, lengths, and constraints as needed

The generated class structure follows this pattern:

```typescript
import type { LocaleType } from "@ooneex/translation";
import { random } from "@ooneex/utils";
import { Column, CreateDateColumn, DeleteDateColumn, PrimaryColumn, UpdateDateColumn } from "typeorm";

@Entity({
  name: "<table_name>",
})
export class <Name>Entity extends BaseEntity {
  @PrimaryColumn({ name: "id", type: "varchar", length: 25 })
  id: string = random.nanoid(25);

  @Column({
    name: "is_locked",
    type: "boolean",
    default: false,
    nullable: true,
  })
  isLocked?: boolean;

  @Column({ name: "locked_at", type: "timestamptz", nullable: true })
  lockedAt?: Date;

  @Column({
    name: "is_blocked",
    type: "boolean",
    default: false,
    nullable: true,
  })
  isBlocked?: boolean;

  @Column({ name: "blocked_at", type: "timestamptz", nullable: true })
  blockedAt?: Date;

  @Column({ name: "block_reason", type: "text", nullable: true })
  blockReason?: string;

  @Column({ name: "is_public", type: "boolean", default: true, nullable: true })
  isPublic?: boolean;

  @Column({ name: "lang", type: "varchar", length: 10, nullable: true })
  lang?: LocaleType;

  @CreateDateColumn({ name: "created_at" })
  createdAt?: Date;

  @UpdateDateColumn({ name: "updated_at" })
  updatedAt?: Date;

  @DeleteDateColumn({ name: "deleted_at" })
  deletedAt?: Date;
}
```

### 4. Complete the test file

Edit `tests/entities/<Name>Entity.spec.ts` to add meaningful tests beyond the scaffolded ones:

- Keep the existing scaffolded tests (class name, id, default columns)
- Add tests for any new entity-specific columns and relations
- Update tests if scaffolded columns were removed

The generated test structure follows this pattern:

```typescript
import { describe, expect, test } from "bun:test";
import { <Name>Entity } from "@/entities/<Name>Entity";

describe("<Name>Entity", () => {
  test("should have class name ending with 'Entity'", () => {
    expect(<Name>Entity.name.endsWith("Entity")).toBe(true);
  });

  test("should have 'id' property with default nanoid", () => {
    const entity = new <Name>Entity();
    expect(entity.id).toBeDefined();
    expect(typeof entity.id).toBe("string");
    expect(entity.id.length).toBe(25);
  });

  test("should have 'isLocked' property", () => {
    const entity = new <Name>Entity();
    expect("isLocked" in entity).toBe(true);
  });

  // ... additional property tests
});
```

### 5. Register the entity in the module

Add the new entity to the module's `entities` array in `src/<PascalModuleName>Module.ts` (e.g., `src/BookModule.ts` for the `book` module):

```typescript
import type { ModuleType } from "@ooneex/module";
import { <Name>Entity } from "./entities/<Name>Entity";

export const <PascalModuleName>Module: ModuleType = {
  controllers: [],
  entities: [<Name>Entity],
  permissions: [],
  middlewares: [],
  cronJobs: [],
  events: [],
};
```

The module file uses PascalCase naming: `<PascalModuleName>Module.ts` with export `<PascalModuleName>Module`.

If the module already has other entities registered, append the new entity to the existing `entities` array and add the import alongside existing imports.

### 6. Create a migration for the entity

After creating or updating an entity, generate a migration to apply the corresponding schema changes to the database.

Run the generator:

```bash
bunx @ooneex/cli@latest make:migration
```

Then read the generated migration file in `src/migrations/` and complete it:

- In the `up` method, write the SQL to create the table (or alter it if updating an existing entity). Include all columns, types, constraints, defaults, and indexes matching the entity definition.
- In the `down` method, write the reverse SQL to undo the changes (e.g., `DROP TABLE` or `ALTER TABLE DROP COLUMN`).
- If the migration depends on another migration (e.g., a foreign key referencing another table), add that migration class to the `getDependencies()` return array.

Example `up` method for a new entity:

```typescript
public async up(tx: TransactionSQL): Promise<void> {
  await tx`
    CREATE TABLE IF NOT EXISTS <table_name> (
      id VARCHAR(25) PRIMARY KEY,
      is_locked BOOLEAN DEFAULT false,
      locked_at TIMESTAMPTZ,
      is_blocked BOOLEAN DEFAULT false,
      blocked_at TIMESTAMPTZ,
      block_reason TEXT,
      is_public BOOLEAN DEFAULT true,
      lang VARCHAR(10),
      created_at TIMESTAMPTZ DEFAULT NOW(),
      updated_at TIMESTAMPTZ DEFAULT NOW(),
      deleted_at TIMESTAMPTZ
    )
  `;
}
```

### 7. Create a repository for the entity

After creating the entity, generate a repository to handle database operations for it.

Run the generator:

```bash
bunx @ooneex/cli@latest make:repository --name <name>
```

Where `<name>` is the same name used for the entity. The command will generate:
- `src/repositories/<Name>Repository.ts` - The repository class
- `tests/repositories/<Name>Repository.spec.ts` - The test file

Then read the generated files and complete the repository implementation:

- Adjust search fields in the `find()` method to match the entity's searchable columns
- Customize relations loading in `findOne`/`findOneBy` if the entity has relations
- Add any domain-specific methods relevant to the entity
- Remove methods that don't apply to the entity
- Update tests to match the final repository methods

### 8. Lint and format

Run linting and formatting on all generated files:

```bash
bunx biome check --fix src/entities/<Name>Entity.ts tests/entities/<Name>Entity.spec.ts src/repositories/<Name>Repository.ts tests/repositories/<Name>Repository.spec.ts src/migrations/
```
