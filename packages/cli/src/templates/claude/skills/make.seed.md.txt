# Make Seed

Generate a new seed file and its test file using the `make:seed` CLI command, then complete the generated code with proper implementation.

## Coding Conventions

- Always explicitly show visibility on class methods and properties (`private`, `public`, `protected`)
- Always use arrow functions when possible, but NOT for class methods (class methods use regular method syntax)
- End type names with the `Type` suffix (e.g., `ExceptionStackFrameType`, `ServiceDataType`)
- Start interface names with the `I` prefix (e.g., `IException`, `IService`, `IAiChat`)
- For Entity class, if property is optional, add `null` to its type.

## Steps

### 1. Run the generator

Run the following command to scaffold the seed file and test files:

```bash
bunx @ooneex/cli@latest make:seed --name=<name> --module=<module>
```

Where `<name>` is the name provided by the user. The `--module` option is optional — if provided, files are generated under `modules/<module>/` instead of the project root. The command will generate:
- `src/seeds/<Name>Seed.ts` - The seed class file (or `modules/<module>/src/seeds/<Name>Seed.ts` with `--module`)
- `src/seeds/<name-seed>.yml` - The seed data file in kebab-case (or `modules/<module>/src/seeds/<name-seed>.yml` with `--module`)
- `tests/seeds/<Name>Seed.spec.ts` - The test file (or `modules/<module>/tests/seeds/<Name>Seed.spec.ts` with `--module`)

It will also:
- Generate a `seeds.ts` root export file in the seeds directory
- Add a `seed:run` script to `package.json` if not already present

### 2. Read the generated files

Read all generated files to understand the scaffolded code.

### 3. Complete the seed data

Edit `src/seeds/<name-seed>.yml` to add the seed data entries. Each entry should have hardcoded nanoid values for `id` fields (generate via `bun -e "import { random } from '@ooneex/utils'; console.log(random.nanoid())"`)

- Do NOT use sequential IDs like `"item-1"`, `"item-2"`
- Ensure the same entity uses the same ID everywhere it appears

### 4. Complete the seed class

Edit `src/seeds/<Name>Seed.ts` to implement:

- Import the relevant entity classes and repository
- Use `resolve()` from `@ooneex/container` to get the repository instance
- Map the imported YAML data to entity instances
- Use the repository to persist the entities

### 5. Complete the test file

Edit `tests/seeds/<Name>Seed.spec.ts` to add meaningful tests beyond the scaffolded ones:

- Keep the existing scaffolded tests (class name, run, isActive, getDependencies, data yml file)
- Add tests relevant to the specific seed behavior

The generated test structure follows this pattern:

```typescript
import { existsSync } from "node:fs";
import { join } from "node:path";
import { describe, expect, test } from "bun:test";
import { <Name>Seed } from "@/seeds/<Name>Seed";

describe("<Name>Seed", () => {
  test("should have class name ending with 'Seed'", () => {
    expect(<Name>Seed.name.endsWith("Seed")).toBe(true);
  });

  test("should have 'run' method", () => {
    expect(<Name>Seed.prototype.run).toBeDefined();
    expect(typeof <Name>Seed.prototype.run).toBe("function");
  });

  test("should have 'isActive' method", () => {
    expect(<Name>Seed.prototype.isActive).toBeDefined();
    expect(typeof <Name>Seed.prototype.isActive).toBe("function");
  });

  test("should have 'getDependencies' method", () => {
    expect(<Name>Seed.prototype.getDependencies).toBeDefined();
    expect(typeof <Name>Seed.prototype.getDependencies).toBe("function");
  });

  test("should have a data yml file", () => {
    const dataFile = join(__dirname, "..", "src", "seeds", "<name-seed>.yml");
    expect(existsSync(dataFile)).toBe(true);
  });
});
```

### 6. Lint and format

Run linting and formatting on the generated files:

```bash
bunx biome check --fix src/seeds/<Name>Seed.ts src/seeds/<name-seed>.yml tests/seeds/<Name>Seed.spec.ts
```
